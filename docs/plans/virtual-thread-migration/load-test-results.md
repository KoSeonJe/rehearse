## 테스트 개요

### 배경

후속 질문 생성 API(`POST /api/v1/interviews/{id}/follow-up`)는 DB 조회 → 외부 AI API 호출(2~3초) → DB 저장으로 구성된다. 원본 코드에서는 이 전체 흐름이 하나의 `@Transactional` 안에 있어, 외부 API 응답을 기다리는 동안 DB 커넥션이 불필요하게 점유되는 구조였다.

### 개선 사항

1. **TX 3단계 분리**: 읽기TX → 트랜잭션 밖 API 호출 → 쓰기TX (커넥션 점유 3초 → 40ms)
2. **Virtual Thread**: Java 21 가상 스레드 + HikariCP 6.2.1 (스레드 200개 상한 제거)
3. **Resilience4j RateLimiter**: 외부 API 호출량 제어

### 테스트 전략

각 테스트는 이전 테스트에서 발견한 병목을 제거하며 단계적으로 진행했다.

| 테스트 | 목적 | 변경한 변수 |
|--------|------|-----------|
| 1 | 현실 운영 환경에서 개선 효과 확인 | TX 분리, VT, RL 8 req/s |
| 2 | RL 제거 후 PT/VT 순수 비교 | RL 제거 |
| 3 | Python mock 병목 제거 후 재비교 | WireMock 교체, pool-size 확장 |
| 4 | PT/VT 처리 한계점 측정 | constant-arrival-rate, pool 100, 3초 지연 |

### 용어 정의

- **실질 처리량**: 성공(2xx) 요청 수 / 테스트 시간. 500 에러 제외.
- **인프라 에러율**: 500번대 서버 에러만 카운트. 400(비즈니스), 429(rate limit) 제외.
- **Peak active DB**: HikariCP에서 동시에 사용 중인 커넥션 수 최대값. pool-size 대비 비율로 포화 여부 판단.
- **Peak pending DB**: 커넥션 풀이 가득 차서 커넥션을 기다리는 요청 수 최대값. 0이면 건강, 높으면 pool 고갈.
- **Peak threads**: JVM 활성 스레드 수 최대값. PT는 Tomcat 200 + 데몬, VT는 가상 스레드 수.
- **k6 필요 VUs**: `constant-arrival-rate`에서 목표 req/s를 유지하기 위해 k6가 생성한 가상 사용자 수. 응답이 느리면 VU가 묶여 더 많은 VU 필요. 값이 클수록 서버 응답이 느리다는 의미.
- **dropped iterations**: k6가 목표 req/s를 보내야 하는데 모든 VU가 응답 대기 중이라 **아예 보내지 못한 요청 수**. 서버가 요청을 소화하지 못하고 있다는 직접적 지표.

## 테스트 1: 현실 시나리오 (RL 8 req/s)

### 목적

GPT-4o Tier 1(500 RPM = ~8 req/s) 기준의 실 서비스 운영 환경에서, TX 분리와 VT 도입의 효과를 단계별로 확인한다.

### 조건

| 항목 | 값 |
|------|-----|
| 외부 API Mock | Python mock (Phase 0: 3초, Phase 1-2: 2초 지연) |
| DB pool-size | 10 |
| RateLimiter | **8 req/s** (GPT-4o Tier 1 기준) |
| 부하 패턴 | k6 ramping-vus: 10→500 VUs (~5분) |
| 테스트 데이터 | 면접 5,000건 |

### 병목 구조 (개선 전)

```
@Transactional 시작 (DB 커넥션 획득)
  → DB 조회 (~20ms)
  → 외부 API 호출 (~3s)  ← 커넥션 점유 중!
  → DB 저장 (~20ms)
@Transactional 종료 (커넥션 반환)
// 총 커넥션 점유: ~3s/요청
// pool-size 10 → 최대 ~3 req/s
```

### 결과

**Phase 0: 원본 코드** — VT 이전(`c1d5f87`), `@Transactional` 내 외부 API 호출, PT 200 스레드

| 지표 | 값 |
|------|-----|
| 성공 | 820 / 9,950 (8%) |
| 인프라 에러율 | **91.74%** |
| 실질 처리량 | **2.8 req/s** |
| Peak active / pending | **10 (상시 포화)** / **190** |
| Peak threads | 215 |

**Phase 1: TX 분리 + RL 8 req/s** — 커넥션 점유 3초 → 40ms

| 지표 | Phase 0 → Phase 1 |
|------|-------------------|
| 성공 | 820 → **2,191** (2.7배) |
| 인프라 에러율 | 91.74% → **1.48%** (90%p 감소) |
| 실질 처리량 | 2.8 → **7.0 req/s** (2.5배) |
| Peak pending | 190 → **0** (완전 해소) |

**Phase 2: +Virtual Thread + RL 8 req/s** — VT 활성화 + HikariCP 6.2.1

| 지표 | Phase 1 (PT) → Phase 2 (VT) |
|------|----------------------------|
| 실질 처리량 | 7.0 → 7.2 req/s (동일) |
| 인프라 에러율 | 1.48% → 1.75% (동일) |
| Peak threads | 215 → **33** (85% 감소) |

### 인사이트

- **TX 분리가 핵심 개선**: 에러율 91% → 1.5%, 처리량 2.5배 증가. 커넥션 풀 고갈이 근본 원인이었음.
- **VT 효과 미미**: RL 8 req/s가 천장이라 PT 200 스레드가 포화되지 않음. VT 이점은 스레드 수 절감(215→33)에 한정.
- **RL이 병목일 때 VT는 의미 없음**: 외부 API 한도가 낮으면 스레드 모델과 무관하게 처리량이 제한됨.

## 테스트 2: RL 제거 — Python Mock

### 목적

RL 8 req/s가 병목이 되어 PT/VT 차이가 드러나지 않았음. RL을 제거하고 순수하게 PT/VT 동시성을 비교한다.

### 조건

| 항목 | 값 |
|------|-----|
| 외부 API Mock | Python mock (Phase 0: 3초, Phase 1-2: 2초 지연) |
| DB pool-size | 10 |
| RateLimiter | **없음** |
| 부하 패턴 | k6 ramping-vus: 10→500 VUs (~5분) |

### 결과

| Phase | 실질 처리량 | 인프라 에러율 | Peak pending | Peak threads |
|-------|-----------|-------------|-------------|-------------|
| 0: 원본 | 2.7 req/s | 91.64% | 190 | 215 |
| 1: TX분리+PT | **31.1 req/s** | 14.93% | 125 | 215 |
| 2: TX분리+VT | 31.0 req/s | 16.03% | 218 | **52** |

### 인사이트

- **TX 분리만으로 11배 처리량 증가** (2.7 → 31.1 req/s). 이 테스트의 가장 큰 발견.
- **PT/VT 처리량 동일** (31 req/s): Python mock 서버가 ~31 req/s에서 병목. `http.server` + `time.sleep()`이 싱글스레드에 가까워 동시 처리 한계.
- **VT pending 증가** (125→218): VT는 Tomcat 200 스레드 제한이 없어 모든 요청을 동시에 받아들임 → DB 커넥션 경쟁 심화.
- **스레드 76% 절감** (215→52)은 확인됨.
- **결론**: Python mock 서버를 교체해야 PT/VT 처리량 차이를 확인할 수 있음.

## 테스트 3: Mock 서버 교체 — WireMock

### 목적

Python mock(~31 req/s)이 병목이었음. WireMock(Jetty 기반, 수천 req/s)으로 교체하여 mock 서버 병목을 제거하고 PT/VT를 재비교한다. 추가로 DB pool-size를 30으로 확장하여 pool 병목도 제거.

### 조건

| 항목 | Pool 10 테스트 | Pool 30 테스트 |
|------|--------------|--------------|
| 외부 API Mock | WireMock Docker (2초 지연) | WireMock Docker (2초 지연) |
| DB pool-size | 10 | **30** |
| RateLimiter | 없음 | 없음 |
| 부하 패턴 | 10→500 VUs (~5분) | **200→500 VUs** (~3분 15초) |

### 결과: Pool-size 10

| 지표 | PT | VT | 변화 |
|------|-----|-----|------|
| 성공 | 2,604 (93%) | 2,523 (92%) | 동일 |
| 인프라 에러율 | 1.65% | 2.30% | 유사 |
| 실질 처리량 | 8.85 req/s | 8.69 req/s | 동일 |
| avg 응답시간 | 18.78s | **16.12s** | **14% 개선** |
| Peak threads | 217 | **32** | **85% 감소** |

### 결과: Pool-size 30

| 지표 | PT | VT | 변화 |
|------|-----|-----|------|
| 성공률 | **100%** | **100%** | 둘 다 에러 0 |
| 실질 처리량 | 9.21 req/s | 9.35 req/s | 동일 |
| avg 응답시간 | 28.8s | **26.52s** | **8% 개선** |
| Peak active DB | 23 / 30 | **13 / 30** | **VT 43% 적게 사용** |
| Peak threads | 217 | **31** | **86% 감소** |

### 인사이트

- **Pool 10→30 확장으로 에러 해소** (1.65% → 0%): pool-size가 충분하면 인프라 에러가 사라짐.
- **처리량은 PT/VT 동일**: `ramping-vus` 방식의 한계. VU가 응답을 기다린 후 재요청하는 구조라, 과부하 시 응답시간이 늘어나면 VU당 요청 수가 줄어 처리량이 수렴함.
- **VT의 커넥션 효율성 확인**: 동일 처리량에서 PT는 23개, VT는 13개 커넥션 사용. VT가 I/O 대기 중 커넥션을 더 빨리 반환.
- **결론**: `ramping-vus`로는 PT/VT 처리량 차이를 측정할 수 없음. 초당 고정 요청을 강제 투입하는 `constant-arrival-rate`가 필요.

## 테스트 4: Constant-Arrival-Rate — PT vs VT 한계 비교

### 목적

이전 테스트들의 한계를 모두 제거하고, PT/VT의 **처리 한계점**을 직접 비교한다.

- Mock 서버 병목 → WireMock 1000 threads로 해소
- DB pool 병목 → pool-size 100으로 해소
- RL 병목 → 제거
- k6 측정 방식 → `constant-arrival-rate`로 초당 요청 수 강제 투입

### 조건

| 항목 | 값 |
|------|-----|
| 외부 API Mock | WireMock Docker (Jetty 1000 threads, **3초 지연**) |
| DB pool-size | **100** |
| RateLimiter | 없음 |
| 테스트 데이터 | 면접 **50,000건** |
| k6 executor | **constant-arrival-rate** |
| 부하 패턴 | 20 → 50 → 80 → 100 → 120 → 150 req/s (각 30초, 총 3분) |

### PT가 66 req/s에서 한계인 이유

```
200 스레드가 동시에 존재하지만, 각 스레드가 외부 API 호출에서 3초간 블로킹됨.

스레드 1:   [----3초 블로킹----][----3초 블로킹----]...
스레드 2:   [----3초 블로킹----][----3초 블로킹----]...
...
스레드 200: [----3초 블로킹----][----3초 블로킹----]...

→ 3초마다 200개 요청 완료 = 200 / 3 = 66 req/s
→ 67번째 요청부터 빈 스레드가 없어 Tomcat 큐 대기
→ 큐 폭증 → 응답시간 증가 → DB 커넥션 연쇄 포화
```

### 결과: Phase 1 — Platform Thread

| 지표 | 값 |
|------|-----|
| 총 요청 | 14,748 |
| 성공 | 5,747 (39%) |
| 성공 avg 응답 | **5.81s** |
| 성공 p95 응답 | **10.11s** |
| Peak active DB | **100 / 100 (포화)** |
| Peak pending DB | **56** |
| Peak threads | **217** |
| dropped iterations | **855** |

### 결과: Phase 2 — Virtual Thread

| 지표 | 값 |
|------|-----|
| 총 요청 | 15,605 |
| 성공 | 6,019 (39%) |
| 성공 avg 응답 | **3.11s** |
| 성공 p95 응답 | **3.55s** |
| Peak active DB | **75 / 100 (여유)** |
| Peak pending DB | **27** |
| Peak threads | **86** |
| dropped iterations | **0** |

### 성공률(39%)이 PT/VT 모두 동일한 이유

`constant-arrival-rate`는 이전 요청의 응답을 기다리지 않고 즉시 다음 요청을 보낸다. k6 스크립트에서 면접 ID를 2회씩 같은 면접에 할당(`Math.floor(__ITER / 2)`)하는데, 두 요청이 동시에 진행되면서 동일한 `orderIndex`로 후속 질문을 insert하려다 unique constraint 충돌(500)이 발생한다.

```
VU 1, iter 0 → interview 1 → follow-up orderIndex=1 생성 중 (3초 소요)
VU 1, iter 1 → interview 1 → follow-up orderIndex=1 동시 생성 시도
→ unique constraint violation → 500 에러
```

이 충돌은 PT/VT 공통으로 ~60% 발생하며, 인프라 병목과 무관한 테스트 데이터의 동시 쓰기 충돌이다. **따라서 응답시간·DB pool 상태·스레드 수가 PT/VT 비교의 핵심 지표.**

### PT vs VT 비교

| 지표 | PT (200 스레드) | VT (무제한) | 변화 |
|------|---------------|-----------|------|
| 총 요청 처리 | 14,748 | **15,605** | VT 6% 더 처리 |
| 성공 | 5,747 | **6,019** | VT 5% 더 성공 |
| **성공 avg 응답** | **5.81s** | **3.11s** | **46% 단축** |
| **성공 p95 응답** | **10.11s** | **3.55s** | **65% 단축** |
| Peak active DB | 100 / 100 (포화) | **75 / 100** | VT 25% 여유 |
| Peak pending DB | 56 | **27** | 52% 감소 |
| **Peak threads** | **217** | **86** | **60% 감소** |
| k6 필요 VUs | 944 | **318** | VT가 1/3 자원으로 동일 부하 |
| dropped iterations | 855 | **0** | VT 누락 없음 |

### 인사이트

**PT가 무너지는 과정:**

```
150 req/s 투입
→ Tomcat 200 스레드, 3초 블로킹 → 최대 66 req/s만 처리 가능
→ 84 req/s가 Tomcat 큐에 적체
→ 대기 중인 스레드들이 DB 커넥션 경쟁 → pool 100/100 완전 포화 (pending 56)
→ 성공 응답시간 5.81s (이론값 3초의 약 2배)
→ 855건의 요청이 아예 처리되지 못함 (dropped)
```

**VT가 안정적인 이유:**

```
150 req/s 투입
→ 가상 스레드 무제한 → 150 req/s 전부 즉시 처리 시작
→ 각 요청이 외부 API 3초 대기 중 캐리어 스레드를 반환
→ DB 커넥션: 요청당 40ms만 점유 후 즉시 반환 → pool 75/100 여유
→ 성공 응답시간 3.11s (이론값 3초에 근접)
→ dropped 0건, 모든 요청을 수용
```

**VT가 동시 요청이 더 많은데 왜 PT보다 DB 부하가 낮은가?**

직관적으로 VT는 스레드 제한이 없어 더 많은 요청이 DB에 몰려야 한다. 하지만 실측은 반대(PT 100/100 포화, VT 75/100 여유). 이유는 **"버스트 vs 균등 분산"** 차이:

```
[PT: 큐잉 → 버스트 패턴]
  200 스레드 꽉 참 → 84 req/s가 Tomcat 큐에 적체
  → 스레드 하나 끝남 → 큐의 요청들이 한꺼번에 스레드 획득
  → 동시에 TX1(DB 조회) 시도 + 끝난 스레드들은 TX2(DB 저장) 시도
  → 커넥션 요청이 "쉬다가 → 한꺼번에 몰림" 패턴으로 반복
  → 순간 burst로 pool 100/100 포화

[VT: 큐 없음 → 균등 분산 패턴]
  스레드 제한 없음 → 150 req/s 전부 즉시 시작, 큐 없음
  → 각 요청이 TX1(20ms) → API(3초) → TX2(20ms) 순서대로 균등하게 흐름
  → 커넥션 사용이 시간축에 고르게 분산 → 순간 burst 없음
  → pool 75/100 유지
```

**VT의 p95 응답시간 3.55초의 의미:**

WireMock 3초 지연 + DB 작업 40ms + 네트워크 오버헤드 = 이론 하한 ~3.05초. p95가 3.55초라는 것은 95%의 성공 요청이 이론값에 0.5초 이내로 처리되었다는 뜻으로, **사실상 병목 없이 동작**하고 있음을 의미. 반면 PT의 p95 10.11초는 이론값의 3.4배로, ~7초가 Tomcat 큐 대기 시간.

**k6 필요 VUs (944 vs 318)의 의미:**

k6가 목표 req/s를 유지하려면 응답 대기 중인 VU 외에 여유 VU가 필요하다. PT는 응답이 느려(avg 5.81s) VU가 오래 묶이므로 944개가 필요했고, VT는 응답이 빨라(avg 3.11s) 318개로 충분했다. 즉 **동일 부하를 VT가 1/3 자원으로 처리**한 셈.

**dropped 855건 (PT)의 의미:**

k6가 150 req/s를 보내야 하는데, 모든 VU(maxVUs)가 응답 대기 중이라 보낼 VU가 없어서 **아예 시도조차 못한 요청**이 855건. 서버가 요청을 소화하지 못하고 있다는 직접적 증거. VT는 0건.

## 종합 결론

### 각 개선의 효과

| 개선 | 핵심 수치 | 필수 여부 |
|------|----------|----------|
| **TX 분리** | 처리량 2.7 → 31.1 req/s (11배), 에러율 91% → 15% | **필수** — 커넥션 풀 고갈 해소 |
| **RateLimiter** | 외부 API 과금/ban 방지 | **필수** — 비즈니스 보호 |
| **Virtual Thread** | 응답시간 46% 단축, 스레드 60% 절감, DB pool 포화 방지 | **필수** — 66 req/s 이상에서 PT 연쇄 장애 |

### 병목 이동 흐름

```
원본: @Transactional 내 외부 API → 커넥션 3초 점유 → pool 고갈 (2.7 req/s)
  ↓ TX 분리
Phase 1: 커넥션 40ms 점유 → 외부 API/스레드가 병목 (31 req/s)
  ↓ Virtual Thread
Phase 2: 스레드 제한 제거 → pool-size가 새로운 병목
  ↓ pool-size 확장
Phase 3: pool 충분 → PT는 200 스레드 천장(66 req/s), VT는 계속 확장
```

### VT 도입 근거

| 근거 | 실측 데이터 |
|------|-----------|
| 응답시간 단축 | PT 5.81s → VT 3.11s (46% 단축, 이론값 근접) |
| DB pool 보호 | PT 100/100 포화 → VT 75/100 여유 |
| 스레드 절감 | PT 217 → VT 86 (60% 감소) |
| 안정성 | PT dropped 855건 → VT 0건 |
| 도입 비용 | `spring.threads.virtual.enabled=true` + HikariCP 6.x |
| 복귀 비용 | 설정 한 줄로 PT 복귀 가능 |
