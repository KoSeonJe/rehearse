# docs/todo — 의사결정 대기 작업 관리

## Purpose

코드를 쓰기 전에 **사용자(또는 팀)의 의사결정이 필요한 작업**을 기록하는 폴더.
"기술 잘 모르는 사람도 5~10분 읽고 결정 가능"한 수준의 배경지식과 옵션 비교를 포함한다.

`.omc/plans/` 와의 차이:
- `.omc/plans/` — 결정이 끝난 작업의 **실행 spec** (개발자용, 짧고 정확)
- `docs/todo/` — 결정 자체가 필요한 작업의 **의사결정 문서** (의사결정자용, 배경지식 포함)

## File Naming

```
docs/todo/todo-YYYY-MM-DD.md         # 단일 의사결정
docs/todo/todo-YYYY-MM-DD-{slug}.md  # 같은 날 여러 건이면 slug 추가
```

예: `todo-2026-04-17.md`, `todo-2026-04-17-cloudfront.md`

## Required Sections

각 todo 문서는 아래 섹션을 **순서대로** 포함한다. 해당 없는 섹션만 삭제 가능 (추가/순서 변경 금지).

```markdown
# {제목 — 결정해야 할 것}

## Status
- 작성일: YYYY-MM-DD
- 상태: Open | Decided | Done | Dropped
- 결정자: @username (결정 후 기입)
- 결정일: YYYY-MM-DD (결정 후 기입)
- 결정 결과: (한 줄 요약 — 결정 후 기입)

## TL;DR
3~5줄. 무엇을 결정해야 하는지 + 권장안 한 줄.

## 왜 이 결정이 필요한가
1~2 문단. 비즈니스 영향(매출/사용자/리스크) 중심.
관련 spec/PR/이슈 링크.

## 배경지식 (의사결정자가 모를 수 있는 것)
**필수 섹션.** 비전공자가 옵션을 이해할 수 있도록 다음을 한 줄~한 문단씩 설명:
- 이 결정과 관련된 **시스템 구성 요소** (다이어그램이나 비유 활용)
- 핵심 **용어** 사전 (CloudFront, OAC, SSG 등 등장하는 모든 약어)
- 현재 **상태** vs 목표 상태
- 비교군의 차이를 가르는 **핵심 개념** (예: SSR vs SSG vs CSR의 차이)

작성 원칙:
- "왜 이게 문제인가"를 먼저 설명 → "그래서 후보가 이런 게 있다"
- 코드 용어 대신 비유 우선 ("CloudFront Function = 식당 앞 안내데스크가 손님을 다른 테이블로 안내")
- 1줄 요약 → 필요 시 펼침

## 옵션 비교

각 옵션마다 동일한 형식으로:

### Option A. {이름}
- **한줄 요약**: 무엇을 어떻게 하는가
- **장점**: 3개 이내
- **단점**: 3개 이내
- **비용**: 시간/돈/복잡도
- **리스크**: 실패 시 영향
- **누가 작업하는가**: 사용자 / 개발자 / 자동화

(B, C, ...)

비교 표는 옵션이 3개 이상일 때만.

## 권장안

**Option {X}** — 한 줄 이유.

### 권장 근거
3~5 bullet. 각 근거는 “이 옵션의 어떤 특성이 우리 상황의 어떤 제약을 푼다” 형식.

### 다른 옵션을 안 고른 이유
- A: 한 줄
- B: 한 줄

## 결정 후 다음 단계
체크리스트로:
- [ ] 사용자가 AWS Console에서 X 작업
- [ ] 개발자가 Y PR 작성
- [ ] CI 검증
- [ ] prod 배포 후 Z 확인

## 참고 자료
- 외부 문서 링크
- 관련 PR/spec 링크
```

## Workflow

1. **작성**: 결정이 필요한 시점에 todo 문서 PR로 추가
2. **검토**: 의사결정자가 읽고 코멘트로 질문 / Resolved 옵션 선택
3. **결정**: Status를 `Decided`로 변경, 결정 결과 기입, 결정자 본인 또는 작성자가 commit
4. **실행**: 결정 결과를 바탕으로 별도 PR로 spec(`.omc/plans/`) 또는 코드 PR 작성
5. **완료**: 실행 PR 머지 후 todo 문서 Status를 `Done`으로 변경, archive 폴더로 이동(`docs/todo/archive/YYYY/`)

## DO / DON'T

**DO**
- 배경지식 섹션을 빼먹지 않는다 (의사결정자가 5분 안에 따라올 수 있게)
- 권장안과 권장 근거를 분리한다 ("내 생각"과 "왜 그렇게 생각하는지"를 구분)
- 결정 후 다음 단계를 체크리스트로 명시한다
- 1개 todo는 1개 결정만 다룬다 (얽히면 분리)

**DON'T**
- 코드 자세히 쓰지 않는다 (그건 spec의 역할)
- 옵션 4개 이상 나열하지 않는다 (선택 피로 ↑) — 진짜 후보만
- 작성자의 권장 없이 옵션만 던지지 않는다 (의사결정자 부담을 떠넘기는 것)
- “일단 다 하자” 식 권장 금지 — 항상 trade-off 명시
