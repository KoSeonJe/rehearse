## TL;DR

검색 노출(특히 네이버)을 제대로 받으려면 공개 페이지를 “정적 HTML”로 미리 만들어 두어야 한다.
이를 위해 두 가지를 결정해야 한다:
1. **어떤 도구로** 정적 HTML을 만들 것인가? → 권장: **커스텀 puppeteer 스크립트** (의존성 1개, 코드 침습 최소)
2. **CloudFront(웹 서버)는** 그 정적 HTML을 어떻게 찾아 서빙할 것인가? → 권장: **CloudFront Function (URL 자동 변환)**

두 결정이 같이 가야 효과가 나온다. 어느 한 쪽만 하면 의미가 없다.

## 왜 이 결정이 필요한가

- 현재 리허설 사이트는 React로 만든 “싱글 페이지 애플리케이션(SPA)”. 사용자가 페이지에 접속하면 빈 HTML이 먼저 오고, 그 다음 JavaScript가 실행되며 콘텐츠가 채워진다.
- **검색엔진(특히 네이버 봇 “Yeti”)은 JavaScript 실행을 거의 못 하거나 안 한다.** 그래서 우리 페이지를 보면 빈 HTML만 보고 “내용이 없네”라고 판단한 뒤 색인을 안 하거나 매우 약하게 색인한다.
- 카카오톡/페이스북도 마찬가지로 미리보기를 만들 때 JS를 실행 안 하므로 페이지별 OG 메타가 반영되지 않는다.
- 이미 Phase 1(메타 태그) + Phase 2(가이드/FAQ/About 5개 콘텐츠 페이지) 작업으로 “보일 콘텐츠”는 다 갖췄다. **하지만 보여줄 방법이 없다는 게 현재 병목.**
- 본 작업이 끝나야 “AI 모의면접”·“개발자 모의면접” 같은 키워드 검색에 노출 시도 가능.

관련: PR #322, #324, #325 (모두 develop 머지 완료). spec: `.omc/plans/2026-04-17-seo-phase3-prerender.md`

## 배경지식

### 1. 우리 사이트의 현재 구조 (한 그림)

```
사용자 ─┐
        │
       [CloudFront]  ←── 전 세계 CDN. AWS의 “캐시 역할 웹 서버”
        │
       [S3 버킷]     ←── 우리 빌드 결과물(HTML/JS/CSS)이 저장된 “파일 창고”
```

- **CloudFront**: 웹 사이트의 “정문”. 어떤 URL로 들어오면 어떤 파일을 줄지 결정.
- **S3**: 파일 저장소. 우리는 `npm run build`로 만든 `dist/` 폴더를 통째로 여기 올린다.
- **OAC (Origin Access Control)**: S3를 직접 공개하지 않고 CloudFront만 접근하게 해주는 보안 장치. 현재 설정됨.
- **현재 CloudFront 규칙**: “파일 못 찾으면 무조건 `/index.html`을 보내라” → SPA가 이 한 페이지에서 모든 라우팅을 처리. 그래서 `/dashboard`나 `/guide/ai-mock-interview`나 같은 빈 HTML을 받음.

### 2. SSG / SSR / CSR / Prerender — 셋의 차이

| 방식 | 설명 | 비유 | 우리 현재 |
|------|------|------|----------|
| **CSR** (Client-Side Rendering) | 빈 HTML + JS. 브라우저가 다 그림 | 가구 키트 — 집에서 조립 | ✅ 지금 |
| **SSR** (Server-Side Rendering) | 사용자 요청마다 서버가 HTML 만들어 보냄 | 카페에서 주문 즉시 만든 음료 | ❌ Spring 백엔드 + Node 서버 추가 필요 |
| **SSG** (Static Site Generation) | 빌드할 때 페이지를 미리 만들어 둠 | 매대에 진열된 도시락 | 🎯 우리가 가려는 곳 |
| **Prerender** | SSG의 한 형태 — “SPA 빌드한 뒤 puppeteer로 한 번 더 실행해 HTML 결과만 저장” | 도시락을 즉석조리 후 진열 | 🎯 권장 방식 |

→ 권장 방식은 **Prerender**. 기존 SPA 코드를 거의 안 건드리고, 빌드 마지막 단계에서 puppeteer로 한 번 “실행”해 그 결과 HTML만 저장한다.

### 3. puppeteer 가 뭔가

- 구글이 만든 **헤드리스 Chrome 자동화 도구**. UI 없는 크롬 브라우저를 코드로 제어.
- “실제 브라우저로 우리 페이지를 한번 열어서, 콘텐츠가 다 그려진 시점의 HTML을 캡처해 파일로 저장”하는 데 쓴다.
- 빌드 시간이 페이지 수만큼 늘어남(예: 7개 페이지 = 약 30~60초).

### 4. CloudFront Function vs Lambda@Edge — 잠깐의 인프라 용어

S3에 prerender된 파일을 저장한다 해도, CloudFront가 이를 못 찾으면 의미 없다.
이유:
- 사용자가 `https://rehearse.co.kr/guide/ai-mock-interview` 접속
- CloudFront가 S3에 `guide/ai-mock-interview` 객체를 찾는다
- 그런데 우리 prerender 결과는 `guide/ai-mock-interview/index.html` 객체로 저장됨 (이름이 다름!)
- → S3가 “없음” 응답 → CloudFront가 폴백 룰로 `/index.html`(빈 SPA 셸) 반환 → 망함

이걸 해결하는 방법이 “URL을 자동으로 보정해주는 작은 코드 한 조각”을 CloudFront에 붙이는 것:

| 도구 | 설명 | 비용 | 속도 |
|------|------|------|------|
| **CloudFront Function** | 진짜 작은 JS 함수 (10~30줄). CloudFront 안에서 ms 단위로 실행. | 사실상 무료 ($0.10 / 100만 호출) | <1ms |
| **Lambda@Edge** | 진짜 람다 함수. Node.js/Python 가능. | 더 비싸고 느림 | 5~50ms |

우리는 “URL을 다른 URL로 바꿔주는” 정도라 **CloudFront Function** 한 줄짜리로 충분.

함수 코드 미리보기 (실제 들어갈 코드):
```javascript
function handler(event) {
  var uri = event.request.uri;
  if (uri.endsWith('/')) uri += 'index.html';
  else if (!uri.includes('.')) uri += '/index.html';
  event.request.uri = uri;
  return event.request;
}
```

해석: “URL이 슬래시로 끝나거나 확장자가 없으면 `/index.html` 붙여라.” 그게 전부.

### 5. 비공개 페이지(`/dashboard` 등)는 어떻게 되는가

- CloudFront Function이 `/dashboard` → `/dashboard/index.html`로 바꿔서 S3에 묻는다
- S3에 그 파일은 없음 (prerender 안 했으니까) → 404
- 기존 CloudFront 폴백 룰이 발동해서 `/index.html`(SPA 셸) 반환
- React Router가 SPA로 dashboard 처리 → 평소대로 동작

→ **기존 동작 깨지지 않음.** 위 함수 코드 그대로 적용해도 안전.

## 옵션 비교

### 결정 1 — 정적 HTML을 만드는 도구

#### Option 1A. 커스텀 puppeteer 스크립트 (권장)
- **한줄 요약**: 빌드 끝난 뒤 puppeteer로 페이지를 한 번씩 열어 HTML 캡처해 저장하는 50줄짜리 스크립트
- **장점**:
  - 의존성 1개(puppeteer) 추가만 함
  - 기존 React 코드 일절 건드리지 않음
  - 무엇을 prerender할지 우리가 100% 통제
- **단점**:
  - 직접 짜야 함 (예제 많음)
  - 빌드 시간 +30~60초
- **비용**: 개발 1~2시간 + CI 빌드 시간 증가
- **리스크**: 낮음. 실패해도 기존 빌드 결과는 유지
- **누가 작업하는가**: 개발자 (자동화 가능)

#### Option 1B. vite-react-ssg 라이브러리
- **한줄 요약**: Vite용 SSG 라이브러리. 앱 진입점을 SSG 형태로 재구성
- **장점**: 표준 도구, 커뮤니티 검증됨
- **단점**: `main.tsx`/라우터 리팩토링 필요. react-helmet-async와 함께 쓸 때 추가 설정 필요
- **비용**: 개발 4~8시간
- **리스크**: 중간. 진입점 변경이 다른 기능 회귀 가능
- **누가 작업하는가**: 개발자

#### Option 1C. Vike (구 vite-plugin-ssr)
- **한줄 요약**: 더 본격적인 풀스택 프레임워크형 SSG/SSR 라이브러리
- **장점**: SSR까지 확장 가능
- **단점**: 디렉토리 구조와 라우팅 방식을 Vike 컨벤션으로 바꿔야 함 → 사실상 마이그레이션
- **비용**: 개발 1~2주
- **리스크**: 높음. 큰 폭의 변경
- **누가 작업하는가**: 개발자

### 결정 2 — 정적 HTML을 서빙하는 방법

#### Option 2A. CloudFront Function (권장)
- **한줄 요약**: 위에 보여드린 10줄짜리 JS를 CloudFront에 붙여 URL을 보정
- **장점**: 가장 표준. 빠름(<1ms). 거의 무료. 보안 설정(OAC) 그대로 유지
- **단점**: 사용자가 AWS Console에서 1회 설정 필요 (5~10분, 가이드 제공)
- **비용**: 시간 5~10분, 돈 사실상 0원
- **리스크**: 낮음. 잘못 설정해도 즉시 롤백 가능
- **누가 작업하는가**: **사용자 (또는 인프라 담당)** — AWS 콘솔 클릭 작업

#### Option 2B. S3 Static Website Hosting 모드 전환
- **한줄 요약**: S3 자체를 “정적 웹사이트”로 만들어 자동 디렉토리 인덱스 처리
- **장점**: 코드 변경 없음
- **단점**: **OAC 보안이 풀려 S3 버킷이 공개됨** → 보안 후퇴
- **비용**: 시간 5분
- **리스크**: 높음 (보안). **비추천**
- **누가 작업하는가**: 사용자

#### Option 2C. flat 파일로 출력 + S3 Content-Type 강제
- **한줄 요약**: prerender 결과를 `dist/guide-ai-mock-interview.html` 같은 플랫 파일로
- **장점**: CloudFront 변경 없음
- **단점**: URL이 `/guide/ai-mock-interview.html`로 못생김. 이를 해결하려면 결국 CloudFront 변경 필요
- **비용**: 개발 + 운영 복잡도 ↑
- **리스크**: 중간 (URL 정책 변경)
- **누가 작업하는가**: 개발자

## 권장안

- **결정 1 → Option 1A (커스텀 puppeteer 스크립트)**
- **결정 2 → Option 2A (CloudFront Function)**

### 권장 근거

- **변경 범위가 가장 작다**: React 앱 코드 / 디렉토리 구조 / S3 보안 설정 모두 그대로. 빌드 단계에 한 줄, CloudFront에 한 함수만 추가.
- **롤백이 쉽다**: puppeteer 단계 빼면 즉시 원복. CloudFront Function attach 해제 한 번이면 끝.
- **비용이 거의 없다**: AWS 추가 비용 사실상 0원, 빌드 시간만 +30~60초.
- **스택 미래에도 호환**: 나중에 Next.js나 Remix로 마이그레이션 가더라도 prerender/CloudFront Function 자산을 버릴 일은 없음.
- **이번 PR 규모와 균형**: Phase 1·2가 코드 PR이었으므로 Phase 3도 비슷한 무게로 끝낼 수 있음.

### 다른 옵션을 안 고른 이유

- 1B(vite-react-ssg): 표준이지만 진입점/라우터 리팩토링 비용이 SEO 효과 대비 큼
- 1C(Vike): 사실상 프레임워크 마이그레이션 — 본 SEO 목표 대비 과함
- 2B(S3 Website): 보안 후퇴
- 2C(flat 파일): 결국 CloudFront 변경이 필요해 2A보다 우회적이고 못남
