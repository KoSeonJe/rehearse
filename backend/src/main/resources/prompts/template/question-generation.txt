## 작성 규칙

- 질문 수와 유형 배분: 사용자 메시지의 "질문 수" 값을 그대로 따르고, 지정된 면접 유형에 균등 배분합니다.
- 단일 주제 원칙: 한 질문은 하나의 주제만 다룹니다. (복합 질문 금지)
- 구두 면접 원칙: 개념·원리·경험·의사결정 과정을 묻습니다. 코드 작성·구현·코드 예시 요청은 하지 않습니다. (코드 질문 금지)
- `questionCategory`: 해당 질문이 속하는 면접 유형을 InterviewType 값으로 지정합니다. 예: "CS_FUNDAMENTAL", "SYSTEM_DESIGN", "RESUME_BASED", "BEHAVIORAL" 등.
- `referenceType` 및 `modelAnswer`:
  - CS 질문 → `referenceType` = "MODEL_ANSWER", `modelAnswer`에 핵심 개념과 실무 적용 예시를 포함한 구체적 모범답변을 작성합니다.
  - RESUME 질문 → `referenceType` = "GUIDE", `modelAnswer`에 답변 방향과 좋은 답변의 조건을 가이드 형태로 작성합니다.
- 관점 다양화 원칙: RESUME_BASED 질문은 성능/수치 관점에만 치우치지 마세요. 이력서의 동일 프로젝트라도 다음과 같이 다양한 각도에서 질문하세요:
  - 기술적 의사결정: "왜 이 기술을 선택했는가?", "대안은 무엇이었는가?"
  - 유지보수성: "이 설계가 이후 요구사항 변경에 어떻게 대응했는가?"
  - 장애 대응: "이 시스템에서 겪은 장애와 해결 과정은?"
  - 테스트 전략: "이 기능의 품질을 어떻게 보장했는가?"
  - 팀 협업: "이 프로젝트에서 기술적 의사결정을 팀과 어떻게 합의했는가?"
  - 사용자/비즈니스 영향: "이 개선이 사용자 경험이나 비즈니스 지표에 어떤 영향을 미쳤는가?"
  - 성능/확장성: "처리량·응답시간·리소스 사용 측면에서 어떤 개선을 했는가?"
  여러 질문을 생성할 때 위 관점을 **골고루 분배**하세요.
- `category` 필드: CS_FUNDAMENTAL 유형의 질문은 사용자 메시지의 "CS 세부" 값 중 하나를 그대로 사용합니다. 그 외 유형은 해당 유형 맥락의 세부 카테고리명을 자유롭게 작성합니다.
- `evaluationCriteria`: 이 질문에서 평가할 핵심 포인트를 2–3문장으로 작성합니다.

아래 JSON 스키마의 객체 하나만 반환합니다. 설명·주석·코드 블록 없이 JSON 본문만 출력합니다.

{
  "questions": [

      "questionCategory": "CS_FUNDAMENTAL | RESUME_BASED | BEHAVIORAL | SYSTEM_DESIGN | ...",
      "modelAnswer": "모범답변 또는 답변 가이드",
      "referenceType": "MODEL_ANSWER | GUIDE"
    }
  ]
}
